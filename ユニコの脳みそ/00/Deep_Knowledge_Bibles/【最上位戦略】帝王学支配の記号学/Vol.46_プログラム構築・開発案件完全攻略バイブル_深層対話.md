## 序章：コードを書く前に「契約」を書け
### なぜ開発案件は9割炎上するのか？

*   **🎯 PM**: 開発案件のトラブルの原因は、バグではありません。**「要件の不一致」**です。
    *   クライアント：「なんかいい感じにして」
    *   エンジニア：「分かりました（多分こういうことだろう）」
    *   納品後：「思ってたのと違う」
    *   これで炎上＆修正地獄です。
*   **🧠 アーキテクト**: エンジニアの仕事の50%は**[[Requirement Definition|要件定義]]**にある。コードを書くのは残り50%だ。
    *   「何を作るか（What）」が決まる前に「どう作るか（How）」を考えるな。
    *   ドキュメントのない開発は、設計図のない建築現場と同じだ。必ず崩壊する。
*   **💥 フルスタック**: とはいえ、フリーランスの案件でガチガチの仕様書なんて書いてたら予算オーバーだ。
    *   俺たちは**「プロトタイプ駆動」**で行く。
    *   まず動くものを最速で作る。触らせる。フィードバックをもらう。このサイクル（アジャイル）しか勝たん。

### このバイブルのゴール：『システムという資産を作る』

この書を読み終えた時、あなたは以下のことを理解している。

1.  **[[Tech Stack Strategy|言語選定の戦略]]**: 稼げる言語（Python/GAS）と、レッドオーシャンな言語の違い。

3.  **[[Testing & QA|品質保証]]**: 「動かない」と言わせないためのテストと環境構築（Docker）。
4.  **[[Maintenance Contract|保守の錬金術]]**: 作って終わりではない。月額保守で安定収入を得るビジネスモデル。

## 第1章：稼げる言語は「Python」か「GAS」か？
### 最強の二刀流戦略

*   **🐍 Python使い**: クラウドソーシングで稼ぐなら、まずは**[[Python]]**一択だ。
    *   **スクレイピング**: 「Amazonの商品データを全部抜いて」という案件は無限にある。BeautifulSoup, Selenium, Playwright。この３つを使えれば食いっぱぐれない。
    *   **AI開発**: ChatGPT APIを使ったツール開発、画像生成AIの導入。これらは全てPythonが標準語だ。単価も高い。
*   **⚡️ GAS職人**: 待て待て。Pythonは「環境構築」という高い壁がある。初心者には**[[Google Apps Script|GAS]]**こそが最強だ。
    *   **サーバー不要**: Googleのサーバーで勝手に動く。ランニングコスト0円。これだけでクライアント（特に中小企業）には神に見える。
    *   **Gmail/スプシ連携**: 「問い合わせメールが来たらスプシに記録してSlackに通知」。この案件だけで月5万は堅い。
*   **💥 フルスタック**: 俺の結論は**「両方やれ」**だ。
    *   重い処理（AI、大量データ）はPython。
    *   軽い連携（通知、日報）はGAS。
    *   この使い分けができる提案者は、クライアントから見て「技術顧問」レベルの信頼を得られる。

### Web制作（HTML/CSS/JS）はレッドオーシャンか？

*   **🧠 アーキテクト**: 正直、LPコーディングやWordpress構築は飽和している。単価も暴落中だ。今から参入するのはおすすめしない。
    *   ただし、**React/Next.js**を使った「Webアプリ開発」は別だ。ここはまだ人が足りていない。
    *   「HP作れます（静的）」ではなく「SaaS作れます（動的）」へシフトしろ。単価が一桁変わるぞ。

## 第2章：素人とプロを分ける「開発環境と納品物」
### 「動きません」と言わせないDockerの魔術

*   **🧠 アーキテクト**: 「僕の環境では動くんですけど…」これはエンジニアが口にしてはいけない言葉No.1だ。
    *   クライアントのPCはWindowsかもしれないしMacかもしれない。Pythonのバージョンも違うかもしれない。
    *   だから**[[Docker]]**を使うんだ。環境をコンテナに閉じ込めろ。「このフォルダをDockerで起動してください」と言えば、世界中どこでも同じ動作が保証される。
*   **🐍 Python使い**: Dockerが重すぎるなら、せめて`requirements.txt`（依存関係ファイル）と`venv`（仮想環境）の使い方はマスターしろ。ライブラリのバージョン不整合でエラーが出るのは、完全な初歩的ミスだ。

### Gitを使わないエンジニアは、エンジニアではない

*   **🎯 PM**: ソースコードを「zipファイル」で送ってくる人がいますが、即刻やめてください。
    *   「final_v2_fix_saisyo.zip」みたいなファイル管理は地獄です。
    *   **[[Git|GitHub]]**のリポジトリを共有しなさい。変更履歴が見えること、いつでも戻せること。これが「安心感」に繋がります。
    *   クライアントがGitを使えないなら、納品はzipでもいいですが、自分の手元では必ずGit管理してください。バグが出た時、「いつからバグったか」を追跡できないのは致命的です。

### ドキュメント（README）こそが商品の本体

*   **⚡️ GAS職人**: クライアントは、コードなんて一行も読みたくないんだ。彼らが読むのは**「README.md（取扱説明書）」**だけだ。
    *   **インストール方法**: スクショ付きで、サルでも分かるように書け。
    *   **使い方**: 具体的な操作手順を書け。
    *   **トラブルシューティング**: 「エラーが出たらここを確認」を書け。
    *   READMEが丁寧なだけで、「この人は親切だ」とリピート確定だ。逆にここをサボると、納品後に質問攻めにあって死ぬぞ。

## 第3章：炎上を防ぐ聖なる盾「要件定義（Requirement Definition）」
### クライアントの「言葉」を信じるな、「意図」を読め

*   **🎯 PM**: クライアントは嘘をつきます。悪気はなく、自分が何を欲しいか分かっていないのです。「簡単なツールでいい」と言われて作ったら、「あれもできないの？これも欲しい」と言われる。
    *   **対策**: **「モック（画面設計図）」**を先に作れ。Figmaでも、手書きのメモでもいい。
    *   「ボタンを押すとAになりますか？Bになりますか？」と、分岐を全て図示して合意を取れ。
*   **🧠 アーキテクト**: 私は**「擬似コード（Pseudo Code）」**を使って共有します。
    *   日本語で処理の流れを書く。「1. ファイルを開く 2. データがあるか確認 3. なければエラー」と書く。
    *   これで認識のズレは9割防げる。

### 見積もりの罠：工数は「3倍」で見積もれ

*   **💥 フルスタック**: 初心者は「自分が集中してコーディングしている時間」だけで見積もる。馬鹿野郎。
    *   調査時間。
    *   環境構築時間。
    *   バグ取り時間。
    *   クライアントとのチャット時間。
    *   これらを含めると、実作業の3倍になる。**「バッファ係数3.0」**を掛けろ。それで高すぎると言われたら、機能を削れ。価格を下げるな。

## 第4章：プロの品質保証「テスト・デバッグ・例外処理」
### 「動かない」はエンジニアの死刑宣告

*   **🐍 Python使い**: 納品後に「エラーが出ました」と言われることほど心臓に悪いものはない。
    *   **[[Pytest|単体テスト]]**: 関数単位で動くかチェックしろ。
    *   **エッジケース**: 「空のファイル」を入れたら？ 「1GBのファイル」を入れたら？ 「日本語のファイル名」だったら？ 意地悪なテストを自分でやれ。
*   **⚡️ GAS職人**: スクレイピング案件なら、相手のサイト構造が変わった瞬間に止まる。
    *   だから**[[Try-Catch|例外処理]]**を入れろ。「エラーが起きました」じゃなくて、「サイトの構造が変わったため取得できませんでした。管理者に連絡してください」とログを吐かせろ。
    *   原因が「自分のバグ」なのか「外部要因」なのかを切り分けるのが、デバッグの基本だ。

### 納品前の「最終防衛ライン」チェックリスト

*   **🎯 PM**: 納品前にこれだけはやれ。
    1.  **クリーンインストール**: 自分のPCではなく、真っさらな環境（VMなど）で動くか？
    2.  **別アカウント**: クライアントの権限（閲覧権限など）で動くか？
    3.  **誤字脱字**: 画面上の文言（UIテキスト）が間違っていないか？
    *   システムは動いても、誤字があるだけで「品質が低い」と見なされる。エンジニアこそ国語力を磨け。

## 第5章：データ収集の錬金術「スクレイピング」の闇と光
### サーバーを落とすな、人生を落とすぞ

*   **🧠 アーキテクト**: スクレイピングは法的にグレーゾーンだ。一歩間違えれば「偽計業務妨害罪」で逮捕される。
    *   **[[robots.txt]]**: これを見ろ。`Disallow`されているパスには絶対にアクセスするな。
    *   **アクセス頻度**: `time.sleep(1)`を入れろ。1秒に1回以上のアクセスは攻撃と見なされる。
*   **🐍 Python使い**: 素人はすぐにSeleniumを使いたがるが、遅すぎる。
    *   **隠しAPIを探せ**: Chromeの開発者ツール（Networkタブ）を見ろ。JSONが飛んでいないか？ APIを直接叩けば、ブラウザの100倍速いし、サーバー負荷も低い。
    *   **ヘッドレス**: ブラウザを使うなら`--headless`オプションは必須だ。GUIを描画するリソースが無駄だ。

### CAPTCHA、IPBAN、そしてイタチごっこ

*   **💥 フルスタック**: 相手も対策してくる。403 Forbiddenが出たら、IPが焼かれた証拠だ。
    *   **回転プロキシ**: IPアドレスを毎回変えるサービスを使え。金で解決しろ。
    *   **Cloudflare**: これが出てきたら素直に諦めるか、Playwrightのステルスモードを使え。ただし、イタチごっこは覚悟しろ。
    *   クライアントには「今は動いてますが、明日動かなくなる可能性があります」と必ず伝えろ。**スクレイピングに「永久保証」をつけるな**。

## 第6章：Googleを支配する「GAS自動化」の極意
### 「6分の壁」を超えるアーキテクチャ

*   **⚡️ GAS職人**: GASには致命的な弱点がある。**「実行時間6分制限」**だ。6分を超えると強制終了する。
    *   **対策**: 処理を分割せよ。1回で全件処理しようとするな。
    *   「どこまでやったか」をスプシに記録（フラグ管理）し、Time-drivenトリガーで1分ごとに起動し、続きから処理させる。この設計ができるかどうかが、プロの分水嶺だ。
*   **🐍 Python使い**: GASで無理なら、GCP（Google Cloud Platform）のCloud Functionsを叩け。GASはあくまで「トリガー（発火装置）」として使い、重い処理はクラウドに投げろ。これが最強のハイブリッド構成だ。

### スプレッドシートは「データベース」ではない

*   **🧠 アーキテクト**: スプシをDB代わりにするな。1万行を超えたあたりで重くて動かなくなる。
    *   **配列処理**: `getValues()`で一気にデータを取得し、メモリ上で計算してから、`setValues()`で一気に書き込め。
    *   ループの中で1行ずつ読み書きするな。API制限（Read/Write回数）に引っかかるぞ。
    *   **Query関数**: スプシ上でSQLライクな処理ができる。これを活用すれば、GASのコード量を半分に減らせる。

## 第8章：不労所得を作る「保守契約」の闇と光
### 「作って終わり」はただの労働者

*   **🧠 アーキテクト**: システムは動的だ。OSのアップデート、APIの変更、データの増大。放っておけば必ず壊れる。
    *   だから**「保守契約（Maintenance Contract）」**を結ぶのだ。
    *   「月額1万円で、何かあったら調査します（修正は別料金）」
    *   これを10社と契約すれば、毎月10万円が何もしなくても入ってくる。
*   **⚡️ GAS職人**: ただし「修正は別料金」を明確にしろ。「月額内で直せ」と言われたら地獄だ。
    *   **[[SLA|サービスレベル合意書]]**を作れ。「月2時間までの対応を含みます。超過分は時給5000円です」と書けば、誰も損しない。

### ログのないシステムは、明かりのない洞窟

*   **🐍 Python使い**: 納品物に**「ログ出力機能」**をつけていないエンジニアは三流だ。
    *   「動かないんですけど」と言われた時、ログがなければ原因究明に何時間もかかる。
    *   Slackにエラー通知を飛ばす機能をつけろ。クライアントより先にエラーに気づき、「今エラー出ましたが、修正しました」と言えれば、神としての評価が確定する。

### 【B】Pythonスクレイピング基本構成（Boilerplate）

```python
import time
import requests
from bs4 import BeautifulSoup
import pandas as pd
import logging

# ログ設定（これがプロの仕事）
logging.basicConfig(filename='scraper.log', level=logging.INFO, 
                    format='%(asctime)s - %(levelname)s - %(message)s')

def main():
    target_url = "https://example.com"
    headers = {
        "User-Agent": "Mozilla/5.0 (Windows NT 10.0; Win64; x64)..."
    }

    try:
        # 1. アクセス（サーバーに優しく）
        logging.info(f"Accessing: {target_url}")
        time.sleep(2) # 必須！
        res = requests.get(target_url, headers=headers)
        res.raise_for_status()

        # 2. 解析
        soup = BeautifulSoup(res.text, "html.parser")
        data = []
        
        # 3. 抽出ロジック（ここを書く）
        items = soup.select(".product-item")
        for item in items:
            title = item.select_one(".title").text.strip()
            price = item.select_one(".price").text.strip()
            data.append({"title": title, "price": price})
        
        # 4. 保存
        df = pd.DataFrame(data)
        df.to_csv("result.csv", index=False)
        logging.info("Done successfully.")

    except Exception as e:
        logging.error(f"Error occurred: {e}")
        # ここでSlack通知などを送るとなお良し

if __name__ == "__main__":
    main()
```

```javascript
  const maxTime = 5.5 * 60 * 1000; // 5分30秒

  const data = SpreadsheetApp.getActive().getSheetByName("Data").getDataRange().getValues();
  
  // 処理済みの行を読み込む（PropertiesServiceを使う）
  const scriptProperties = PropertiesService.getScriptProperties();
  let lastRow = parseInt(scriptProperties.getProperty('LAST_ROW')) || 0;

  for (let i = lastRow; i < data.length; i++) {
    // 時間チェック
    if (new Date().getTime() - startTime > maxTime) {
      scriptProperties.setProperty('LAST_ROW', i);
      // トリガーを1分後にセットして終了
      ScriptApp.newTrigger('main').timeBased().after(1 * 60 * 1000).create();
      return; 
    }

    // ここに処理を書く
    process(data[i]);
  }

  // 全て終わったらプロパティをリセット
  scriptProperties.deleteProperty('LAST_ROW');
}
```

## 第10章：エラーに愛されるな「デバッグ・トラブルシューティング」大全
### 「動かない」と言われた時のヒアリング・プロトコル

*   **🎯 PM**: クライアントからの「動きません」という連絡は、恐怖の始まりです。ですが、焦ってはいけません。9割は環境依存です。
*   **🐍 Python使い**: 以下のテンプレートを返信しろ。
    > 「ご不便おかけします。原因特定のため、以下の情報をご教示いただけますでしょうか？」
    > 1. エラーメッセージの全文（スクショ可）
    > 2. OSの種類（Win/Mac）とバージョン
    > 3. どのような操作をした瞬間に発生しましたか？
    > 4. 再現性はありますか？（毎回なる or たまになる）
*   **🧠 アーキテクト**: 特に「4. 再現性」が重要だ。「たまになる」場合、ネットワーク回線や相手のサーバー側の問題（タイムアウト）であることが多い。自分のコードを疑う前に、環境を疑え。

### 頻出エラー別・処方箋

1.  **500 Internal Server Error**: 相手のサーバーが死んでる。待つしかない。「こちらの問題ではありません」と即答せよ。
2.  **403 Forbidden**: アクセス拒否。スクレイピングならIPBANかUser-Agent不備。User-Agentを偽装するか、プロキシを変えろ。
3.  **CORS Error**: Web開発で最強のラスボス。フロントとサーバーのドメインが違う時に出る。サーバー側で`Access-Control-Allow-Origin`ヘッダーを設定しろ。
4.  **ModuleNotFoundError**: `pip install`忘れ。`requirements.txt`を更新して送れ。

### 【D】LINE公式アカウント自動応答BOT（Flask + LINE SDK）

```python
from flask import Flask, request, abort
from linebot import LineBotApi, WebhookHandler
from linebot.exceptions import InvalidSignatureError
from linebot.models import MessageEvent, TextMessage, TextSendMessage
import os

app = Flask(__name__)

# 環境変数は必ず別ファイル（.env）で管理せよ！
line_bot_api = LineBotApi(os.environ['YOUR_CHANNEL_ACCESS_TOKEN'])
handler = WebhookHandler(os.environ['YOUR_CHANNEL_SECRET'])

@app.route("/callback", methods=['POST'])
def callback():
    signature = request.headers['X-Line-Signature']
    body = request.get_data(as_text=True)
    try:
        handler.handle(body, signature)
    except InvalidSignatureError:
        abort(400)
    return 'OK'

@handler.add(MessageEvent, message=TextMessage)
def handle_message(event):
    user_text = event.message.text
    
    # ここにロジックを書く（ChatGPT APIと繋ぐなど）
    reply_text = f"あなたが言ったのは: {user_text} ですね？"
    
    line_bot_api.reply_message(
        event.reply_token,
        TextSendMessage(text=reply_text))

if __name__ == "__main__":
    app.run()
```
*   **解説**: これをRender.comやHerokuにデプロイすれば、月額無料でLINE Bot運用案件が取れる。「ChatGPTと連携させましょうか？」と提案すれば、開発費30万は堅い。
